# 第 6 章：长对话管理

你跟 Claude Code 聊得越久，它就越傻。这不是玄学，是上下文窗口的物理限制。一个中型项目跑 3 轮 Grep + Read 就能吃掉 100K tokens，你以为还有很多空间，其实已经开始丢信息了。

这章讲清楚三件事：窗口到底多大、怎么被压缩的、怎么让关键信息活到最后。

---

## 6.1 上下文窗口到底有多大

三个数字你要记住：

- **Opus 4.6 [1m]**：1,000,000 tokens，约 75 万中文字
- **Sonnet 4.6**：200,000 tokens，约 15 万中文字
- **实际可用** ≠ 窗口大小
- ** Opus 4.7 [340K]

实际可用空间要扣 4 笔"税"：

| 扣除项 | 占用量 | 说明 |
|--------|--------|------|
| System prompt 静态部分 | 5,000 - 10,000 tokens | 7 段固定内容 + 工具说明书，每个用户一样 |
| 工具定义 | 视 MCP 数量而定 | 10 个 MCP 工具大约 5K tokens，30 个就是 15K |
| 压缩预留 | 33,000 tokens | Auto Compact 需要的缓冲区 |
| CLAUDE.md + Memory | 看你写了多少 | 作为第一条 user message 注入 |

算一笔账。Opus 1M 窗口，扣掉 system prompt 8K、10 个 MCP 工具定义 5K、压缩预留 33K、CLAUDE.md 2K，实际可用约 **952K tokens**。听起来很宽裕。

但 Sonnet 200K 窗口，同样扣法，实际可用只有 **152K tokens**。一个中型 React 项目让它读 10 个文件、跑 5 轮 Grep，轻松吃掉 80K。剩下的 72K 要装你所有的对话历史。

> 【截图位置：状态栏 token 计数器截图，标注当前已用/总可用】

关键认知：**Sonnet 用户比 Opus 用户更需要管理上下文。** 你的窗口只有 Opus 的 1/5，但工具定义、system prompt、压缩预留这些固定开销是一样的。相当于一个 200 平的房子和一个 40 平的房子，家具占掉的面积差不多。

## 6.2 五层压缩机制详解

Claude Code 不是等到满了才压缩。它有 5 层递进式压缩，从无感到有感，越往后代价越大。每一层都在上一层不够用时才启动。

### 第 1 层：Tool Result Budget

**每轮循环开始时执行。** 统计所有工具返回结果的累积大小。当累积超过预设阈值时，把超大的结果持久化到磁盘（写入临时文件），对话里只保留一段预览摘要。

举个例子：你让它 Read 一个 5000 行的文件，结果占了 20K tokens。到了下一轮循环，这个结果可能被写入磁盘，对话里只留"该文件已读取，共 5000 行，前 50 行内容为..."这样的预览。

你完全感知不到这个过程。它不会影响后续操作——如果模型需要再看那个文件，它会重新 Read。

### 第 2 层：Snip Compact

**每轮循环执行。** 裁剪掉不再需要的历史片段。比如 3 轮前你让它搜索某个函数，搜到了之后讨论了其他事情，那个搜索结果对当前任务已经没用了，会被裁掉。

判断标准是：这段历史对当前进行中的任务有没有贡献。已经被使用过且不再被引用的工具结果，是第一批被裁的。

### 第 3 层：Microcompact

**每轮循环执行。** 专门压缩大型工具输出中的冗余内容。比如一个 Grep 返回了 200 行匹配结果，其中 150 行结构重复，Microcompact 会把重复的部分合并成"...及其他 150 个类似匹配"。

前 3 层你基本感知不到。信息损失接近零，因为被压缩的要么是可以重新获取的（文件内容），要么是冗余的（重复搜索结果）。

### 第 4 层：Context Collapse

**每轮循环执行，但只在前 3 层不够时才激活。** 把旧的对话段落折叠成摘要。

比如 20 轮前你和它讨论了某个 API 设计方案，Context Collapse 会把那 20 轮对话压缩成一段 200 字的摘要："用户和 Claude 讨论了 XX API 的设计方案，最终决定用 YY 方式实现，关键约束是 ZZ。"

从这一层开始，**细节会丢失。** 那 20 轮对话里你提到的某个边界情况、某个你不喜欢的命名方式，可能就不在摘要里了。

之后你问"我之前说那个 edge case 怎么处理来着"，它可能答不上来。

### 第 5 层：Auto Compact

**token 接近上限时触发。** 这是最重的操作——fork 一个专门的 Agent，让它阅读整个对话历史，输出一份完整的 9 段结构化摘要，然后**用这份摘要替换掉所有历史消息。**

代价很高：多消耗 1 次 API 调用（那个 fork Agent 要读完整个对话再生成摘要）。而且摘要再怎么好也只能覆盖原始对话的一小部分信息。

我踩过一个坑：一个 React 重构任务，聊到 400K tokens 时触发了 Auto Compact。压缩后它完全忘了我之前说的"不要用 useEffect 做数据获取，用 React Query"，结果又写了一堆 useEffect。

从那以后我学会了在 200K 左右就手动 /compact。

### 第 5b 层：Reactive Compact

**收到 prompt-too-long 错误后紧急触发。** 逻辑跟 Auto Compact 一样，区别是这是被动的——已经超了才压缩，而不是快超了提前压缩。

触发 Reactive Compact 说明你连 Auto Compact 的缓冲区都用完了。这种情况下压缩质量更差，因为系统在"紧急状态"下运行，没有足够的空间给 fork Agent 生成高质量摘要。

## 6.3 Auto Compact 的触发阈值

两个公式：

```
effectiveContextWindow = 模型上下文窗口 − 20,000
autoCompactThreshold = effectiveContextWindow − 13,000
```

翻译成人话：**上下文窗口 − 33,000 tokens 时触发自动压缩。**

| 模型 | 窗口 | 触发阈值 |
|------|------|---------|
| Opus 1M | 1,000,000 | ~967,000 tokens |
| Sonnet 200K | 200,000 | ~167,000 tokens |

Sonnet 用户特别要注意——167K 的触发阈值，减去固定开销 50K 左右，你实际能自由使用的空间只有 ~120K。一个复杂 debug session 跑 10 轮工具调用就能吃掉这么多。

**自定义阈值：** 设环境变量 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`，值是百分比。比如设成 70，就是用掉窗口的 70% 时触发。

对 Sonnet 用户来说，把这个值设成 60-70 是个好主意——提前触发，压缩质量更高，因为要压缩的内容少。

```bash
export CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=70
```

## 6.4 Token 警告 4 阶段

Claude Code 不会突然断掉。它有 4 级渐进式预警：

**第 1 阶段：接近阈值。** 距离 autoCompactThreshold 约 20K tokens 时，状态栏出现黄色提示。这时候你还有充足的空间，回复质量不受影响。但这是信号——该考虑压缩了。

**第 2 阶段：严重接近。** 距离阈值更近（不到 10K tokens），警告变红。你可能已经开始感知到回复质量下降——它开始忘东西、重复自己、偶尔答非所问。

这是因为上下文太长导致注意力分散，模型在 1M 个 token 里找关键信息的能力是递减的。

**第 3 阶段：触发自动压缩。** 后台自动 fork 一个 Agent 做摘要，你会看到终端出现 "Compacting..." 字样。这个过程需要 10-30 秒（取决于对话长度）。

完成后对话历史被替换为摘要，token 计数器回到很低的数字。

**第 4 阶段：硬阻塞。** 距离窗口上限仅 3K tokens 时，完全无法继续。输入任何内容都会被拒绝，必须先压缩或新开会话。

这是最后的安全网，到这一步说明 Auto Compact 和 Reactive Compact 都没来得及处理。

到第 3 阶段才反应就晚了。理想状态是在第 1 阶段看到黄色提示时就主动处理。

## 6.5 /compact 的最佳时机

4 个信号告诉你该手动压缩了：

**完成一个阶段性任务后。** 刚写完一个功能、一轮 debug 结束、一次 code review 做完。这时候对话里的工具调用结果、中间讨论已经变成了"历史"，压缩掉不影响后续工作。

而且此时摘要的目标很清晰——"完成了 XX，接下来做 YY"，摘要质量高。

**感觉回复质量明显下降。** 开始重复自己、忘记之前的约定、答非所问、给出明显错误的代码。这些都是上下文过长导致注意力退化的信号。

**准备开始一个新主题。** 你刚讨论完 API 设计，现在要聊 UI 实现。旧的 API 讨论上下文不仅没用，还是噪音。模型可能会被旧上下文干扰，在 UI 代码里混入 API 相关的考虑。

**状态栏出现黄色警告。** 不要等红色。黄色阶段压缩，你有足够的缓冲空间，压缩过程更从容，摘要质量更好。

核心原则：**不要等自动触发。** 自动触发意味着已经很紧了，压缩时可用空间更少，fork 出来的 Agent 能做的摘要也更粗糙。

主动压缩 = 你控制压缩时机 = 你控制哪些信息被保留。

## 6.6 压缩后的摘要结构

Auto Compact fork 出来的 Agent 会输出 9 段结构化摘要：

| 段落 | 内容 | 重要性 |
|------|------|--------|
| Primary Request | 你最初要干什么 | 最高——决定了压缩后的方向 |
| Key Technical Concepts | 涉及的技术概念和框架 | 高 |
| Files and Code Sections | 关键文件路径和代码段 | 高——压缩后还能找到文件 |
| Current Progress | 做到哪了 | 高——知道从哪继续 |
| Remaining Work | 还剩什么没做 | 高 |
| Key Decisions | 做了哪些设计决策 | 中——容易丢细节 |
| Important Constraints | 约束条件 | 中 |
| User Preferences | 你的偏好和习惯 | 中——可能被简化 |
| Critical Details | 不能丢的关键细节 | 高——但"什么是关键"由摘要 Agent 判断 |

这 9 段摘要直接替换掉之前的所有对话消息。压缩后它对之前的记忆，100% 取决于这 9 段写得好不好。

问题在于：摘要 Agent 需要在有限篇幅内覆盖整个对话历史。如果你的对话有 500K tokens，涉及 3 个完全不同的主题，摘要 Agent 很难面面俱到。

这就是为什么短对话多次压缩比一次超长对话再压缩效果好。

## 6.7 减少压缩损失的 4 个操作习惯

### 关键信息放在最近的消息里

压缩优先删远的、保留近的。如果有一个关键决策（比如"这个项目必须用 TypeScript strict mode"），在最近的消息里重提一次。

不需要完整重复，一句"继续保持 TypeScript strict mode"就够了。

这个技巧的原理：Context Collapse 和 Auto Compact 都会给近期消息更高的权重。距离当前越近的信息，被保留在摘要里的概率越高。

### 用 TodoWrite 固定关键进度

TodoWrite 创建的 Todo 列表是独立存储的，不在对话消息流里，不会被任何压缩层吃掉。

把"已完成 XX""待做 YY""约束 ZZ"写进 Todo，等于给记忆上了保险。压缩后 Claude 可能忘了你们讨论的细节，但打开 Todo 就能看到所有关键节点。

实操：每完成一个子任务，让 Claude 更新一次 Todo。不只是打勾，还要写一句总结。

### 分次对话比一次超长对话好

完成一个主题后新开会话，带上必要的上下文重新开始。**10 次 50K 对话的总质量远高于 1 次 500K 对话。**

为什么？每次新对话，system prompt 和 CLAUDE.md 是完整的、没有被压缩的、注意力 100% 集中的。而 500K 对话里，模型的注意力被分散到大量历史信息上。

具体做法：在一个对话里完成一个功能后，新开对话，第一条消息写"我刚完成了 XX 功能（见 src/xxx.ts），现在要做 YY 功能，约束条件是 ZZ"。

这比让模型从 500K 的压缩摘要里找信息靠谱得多。

### 在关键节点手动 /compact

如果任务确实需要长对话（比如复杂的多步骤重构），那每完成一个子任务就手动 /compact 一次。

让摘要保持精炼——每次压缩只需要总结最近一段工作，而不是整个对话的全部历史。

比你在 500K 时才触发一次自动压缩好太多。那时候 9 段摘要要覆盖几十个子任务，每个子任务只能分到几十个字的描述。
